# Memory that knows when to stop believing itself

Every AI memory framework I've seen — Cognee included, before this weekend — is really good at one thing: finding the fact you're looking for. Ask "where does the user live" and vector search will happily hand you back "the user lives in Prague," ranked #1 by cosine similarity. The problem is it'll do that with equal confidence three months after the user told the agent they moved to Berlin.

The [STALE benchmark](https://arxiv.org/abs/2605.06527) (arxiv 2605.06527, May 2026) put a number on this that stuck with me: the best frontier LLM available gets it right only 55.2% of the time, and every memory framework the paper tested — Mem0, Zep, LightMem, A-mem, LiCoMemory — scored under 18%. Not because retrieval is broken. Because retrieval was never the question. Nothing in these systems asks "is this thing I'm about to hand back still true?"

The paper calls this **implicit conflict**: a new observation invalidates something already stored, without ever explicitly saying so. It splits into two flavors. Type I is the easy one — same attribute, new value ("lives in Prague" → "just moved to Berlin"). Type II is the one that actually breaks agents in production: a fact that's never mentioned again quietly stops being true because something upstream of it changed. "The user commutes by bicycle every day" doesn't get contradicted when they say "I broke my leg yesterday" — it just becomes false, silently, forever, unless something is specifically looking for that.

So I built **STALE-Guard** on top of Cognee: a write-time judge that, every time a new fact is remembered, checks it against what's already stored and asks an LLM to classify the relationship — KEEP, STALE, REPLACE, or UNKNOWN — against exactly that Type I / Type II taxonomy. It looks two ways: vector similarity against the new fact's text (which handles Type I fine), and graph traversal — walking the actual entity edges out from the new fact's node, two hops, regardless of what the text says. That second path matters more than it sounds. In a small memory graph, vector similarity accidentally catches most Type II conflicts too, because there's nothing else competing for the top-k slots. I proved to myself this wasn't a coincidence by deliberately building an adversarial test: fifteen filler facts about skiing trips and old injuries, all sharing vocabulary with a "broke my leg skiing" trigger, crowding out a "commutes by bicycle every day" fact that shares zero words with it. Vector search missed it completely. Graph traversal — walking from the new fact's chunk, through the shared "user" entity, to everything else connected to that entity — found it anyway, because the connection was never lexical to begin with.

Before anything gets marked invalid, it goes through one more check: a second, adversarial LLM call whose only job is to try to find a reason the first judge might be wrong. Marking something stale is destructive — it's hidden from every future answer — so a single LLM call getting it wrong once was a risk I didn't want to ship silently. In testing, this refuter is genuinely conservative: on one scenario (a layoff that should invalidate a "wakes at 6am to catch the train" habit), it pushed back with "habits often persist despite losing the reason for them" — which is a real, debatable point, not a hallucination. The system erred toward not suppressing a memory it wasn't sure about, and said so, in an auditable sentence I can point to. That felt like the right trade-off more than a higher accuracy number would have.

Verdicts get written onto the graph nodes themselves (nothing is deleted — Cognee's graph is append-only by design) through a new `stale_state` accessor pair I added to Cognee's `GraphDBInterface`, mirroring the `feedback_weight`/`truth_state` accessors already living there. At read time, a constrained-recall step filters stale nodes out of the answer context and raises a `premise_warning` when your *question itself* assumes something outdated. And because I patched three lines in Cognee's own graph visualizer, you can actually watch it happen — stale nodes render in red, using the same color-override mechanism Cognee already uses for ontology-matched nodes.

Across 13 hand-written conflict scenarios — location, job, relationship, diet, health, life events, and one coding-agent scenario ("we use Redux" quietly invalidated by a migration to Zustand) — plain Cognee and STALE-Guard answered the same questions, graded by an LLM reading full context rather than keyword matching (which I learned the hard way gives false readings: an answer correctly saying "since you divorced Sarah, only list her if you intentionally want her" still contains the word "Sarah"). The gap between the two held up across both conflict types.

The honest gap: `recall()` still answers through its own vector-search-plus-LLM-call rather than hooking into Cognee's actual retrieval stack, so today it's a second path alongside Cognee's search rather than a modification of it. And reproducing the STALE benchmark's actual 1,200-query protocol is the natural next step to turn a demo-grade number into a citable one.

Code, numbers, and the adversarial propagation-proof script are all in the repo.
